# Claude.md

## Project Overview

- Name: `dcraw-wasm`
- Goal: Compile `dcraw.c` to WebAssembly and expose typed JavaScript/TypeScript APIs for RAW camera parsing in browser and Node.js workflows.
- Current status: Publish-ready npm package shape is in place (`exports`, `files`, `prepublishOnly`), with smoke tests and tarball integration tests.

## What This Repository Contains

- Native source: `src/lib/dcraw.c` (upstream dcraw C implementation).
- WASM build output: `bin/dcraw.js` + `bin/dcraw.wasm` (generated by Emscripten via `make dev`/`make prod`).
- Low-level internal wrapper: `src/dcraw-wasm.js` (class-based interface around Emscripten module + MEMFS).
- Public TypeScript API layer: `src/public/*` (facade, metadata parsing, browser/node helpers).
- Compiled public API artifacts: `dist/*` (generated by `npm run build:api`).
- Demo suite entry: `demo/index.html` (links to browser convenience demo, browser RawDecoder demo, and Node usage page).
- Build orchestration: `makefile` + npm scripts in `package.json`.
- Smoke test harness: `test/download-fixture.js` + `test/smoke.test.js`.
- Pack integration test harness: `test/integration-pack.test.js`.

## Core Architecture

1. `emcc` compiles `src/lib/dcraw.c` into modular ES module output (`bin/dcraw.js`, WASM enabled).
2. `src/dcraw-wasm.js` dynamically loads the Emscripten module factory from either `./bin/dcraw.js` (demo copy layout) or `../bin/dcraw.js` (npm package layout), initializes it asynchronously, and binds to Emscripten `FS`.
3. `src/public/raw-decoder.ts` provides the typed `RawDecoder` facade. Each decode method (`readMetadata`, `extractThumbnail`, `analyze`) is async and creates its own fresh `DcrawWasm` instance, uses it for exactly one operation, then discards it. `init()` is optional and stores module options forwarded to all isolated runs. `runInternal()` is an advanced escape hatch that requires `init()` and is explicitly one-shot.
4. `src/public/isolated-analysis.ts` delegates to `RawDecoder.analyze()` directly, since isolation is now handled inside `RawDecoder`.
5. `src/public/metadata.ts` parses raw metadata text into typed/stable metadata properties.
6. `src/public/browser.ts` and `src/public/node.ts` implement convenience layers for browser file/drop flows and Node batch processing.
7. At runtime, RAW file bytes are written to MEMFS, `_runMain` (aliased `main`) is invoked with mapped CLI-like args, and output is read from stdout or generated thumbnail file.

## Build And Run Commands

- Install deps: `npm install`
- Build dev artifacts (WASM + public API): `npm run build` or `npm run build:dev`
- Build optimized artifacts (prod WASM + public API): `npm run build:prod`
- Build only public API TS layer: `npm run build:api`
- Start demo: `npm run start`
  - Builds and copies demo dependencies
  - Copies `bin/*` to `demo/bin/`
  - Copies `src/dcraw-wasm.js` to `demo/`
  - Copies `dist/*` to `demo/public-api/`
  - Launches `emrun` with `demo/index.html` using macOS default browser via `/usr/bin/open`
- Run Node batch demo: `npm run demo:node`
- Lint: `npm run lint`
- Smoke tests: `npm test`
  - Prefers local cache `test/fixtures-local/SAMPLE.ARW` when available.
  - Falls back to downloading `https://rawcameraimages.com/demo/SAMPLE.ARW` when local fixture is missing.
  - Copies fixture into `test/fixtures/SAMPLE.ARW` for test runtime compatibility.
  - Runs metadata + thumbnail extraction tests using `node:test`.
- Tarball integration test: `npm run test:pack`
  - Builds production artifacts.
  - Packs module via `npm pack`.
  - Installs tarball in a temp sandbox and validates one metadata decode.
- Publish gate: `npm run prepublishOnly`
  - Runs `build:prod`, `test`, and `test:pack`.

## Public JS API (Current)

Package exports:

- `dcraw-wasm` (default): typed main API from `dist/index.js`
- `dcraw-wasm/browser`: browser convenience API from `dist/browser.js`
- `dcraw-wasm/node`: Node batch convenience API from `dist/node.js`
- `dcraw-wasm/internal`: low-level wrapper from `src/dcraw-wasm.js`

Main public API (`dcraw-wasm`):

- `new RawDecoder()`
- `await rawDecoder.init(moduleOptions?)` — optional; stores module options (e.g. `locateFile`) forwarded to all isolated runs
- `await rawDecoder.readMetadata(rawBuffer, options?)` — async, isolated runtime per call
- `await rawDecoder.extractThumbnail(rawBuffer, options?)` — async, isolated runtime per call
- `await rawDecoder.analyze(rawBuffer, options?)` — async, isolated runtime per call
- `await rawDecoder.readMetadataSafe(...)` — async safe wrapper
- `await rawDecoder.extractThumbnailSafe(...)` — async safe wrapper
- `rawDecoder.runInternal(rawBuffer, internalOptions)` — sync, requires prior `init()`, one-shot/advanced

Browser convenience API (`dcraw-wasm/browser`):

- `analyzeBrowserFiles(fileListOrArray, options?)`
- `analyzeDroppedFiles(dataTransfer, options?)`

Node convenience API (`dcraw-wasm/node`):

- `analyzeNodeFiles(filePaths, options?)`

## Option Mapping Notes

- `RawDecoder` exposes descriptive options (examples: `brightness`, `interpolationQuality`, `cameraWhiteBalance`) and maps them to internal dcraw option keys.
- Low-level `src/dcraw-wasm.js` option mappings still map to dcraw flags (examples: `identify -> -i`, `verbose -> -v`, `extractThumbnail -> -e`).
- Buffer-type mapping object exists in source but is not currently wired into `run(...)` argument composition.

## Build Profile Notes

- `make all` aliases to `make dev`.
- `dev` profile keeps debug ergonomics (`-O0`, assertions enabled, address sanitizer).
- `prod` profile uses `-O1` and disables assertions (`-s ASSERTIONS=0`).
  - `-O2` and `-O3` currently cause runtime decode instability (`memory access out of bounds`) for this vendored `dcraw.c` build.
- Deprecated `--memory-init-file` was removed.
- Runtime symbol exports now include `_malloc` and `_free` to support Emscripten-5-compatible argument marshaling in the JS wrapper.

## Known Code Risks / Gotchas

- `src/lib/dcraw.c` is vendored upstream code and still emits a few compiler warnings under newer toolchains.
- Buffer-type option mapping object exists but is not currently consumed by `run(...)` argument composition.
- High optimization levels (`-O2`/`-O3`) are not currently release-safe due to runtime crashes; keep `prod` at `-O1` until root-caused.
- `runInternal()` on `RawDecoder` requires `init()` and is explicitly one-shot — the underlying `DcrawWasm` instance is reused from `init()` and cannot safely decode more than once. Document this clearly at call sites.

## Conventions For Future Changes

- Keep public API ergonomics centered on `src/public/*`; keep `src/dcraw-wasm.js` as compatibility/low-level runtime unless intentionally redesigning.
- Each `RawDecoder` decode method (`readMetadata`, `extractThumbnail`, `analyze`) creates a fresh `DcrawWasm` runtime, uses it once, and discards it. Preserve this isolation discipline.
- `init()` stores module options used by all isolated runs — if adding new methods that need module options at runtime, always forward `this.moduleOptions`.
- Preserve MEMFS lifecycle discipline inside `DcrawWasm`:
  - write input file
  - invoke `_runMain`
  - read outputs
  - cleanup temp files
- Keep wrapper stdout callbacks bound to class context (do not pass unbound method references to Emscripten hooks).
- If editing Emscripten exports in `makefile`, keep wrapper calls (`_runMain`, `FS`, allocation helpers) in sync.
- Prefer validating changes via `npm run build`, `npm run test`, and demo checks (`npm run start`, `npm run demo:node`).

## Suggested Task Playbooks For LLMs

### If asked to fix runtime behavior

1. Inspect `src/dcraw-wasm.js` first.
2. Verify option-to-flag mapping.
3. Verify stdout callback binding and output capture.
4. Verify MEMFS read/write/unlink order.
5. Inspect `src/public/raw-decoder.ts` runtime loader behavior if issue reproduces only in demos/package layouts.
6. Rebuild (`npm run build`) and run smoke tests (`npm test`).

### If asked to improve build output

1. Inspect `makefile` flags.
2. Keep `dev` and `prod` profile behavior explicit (debug safety vs optimization).
3. Validate that wrapper-required exported symbols remain available.

### If asked to improve docs

1. Update `README.md` usage sections from demo suite pages and package exports.
2. Keep `Claude.md` and `README.md` in sync on public API names and script names.
3. Document supported option keys from mapping objects in wrapper/facade.
